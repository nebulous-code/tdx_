## Verdict
Strong foundation. **Relational auth scoping is clean** (one chokepoint: `authz.ts` → `denyAccess`; every service query filters `owner_id`; `createLink` checks both endpoints; `getLinksFor` re-checks per far endpoint). **No XSS** (markdown-it `html:false`; all custom rules + the `innerHTML`/`v-html` paths escape user content or are static). **Parity engine untouched** (unified query orchestrates around `Q`); 80/80 tests, tsc + lint clean. No listener leaks, no FTS injection, no reachable traversal in the create path. The issues below are edge cases plus one latent multi-tenant gap.

## Tier 1 — real bugs affecting the single user today — **FIXED (this pass)**
1. **[FIXED] Router ↔ dirty-guard desync on Back/Forward.** `index.html routeFromHash` + `data.js setView`: the browser changes the hash before the guard runs; cancelling "keep editing" left the URL on the new app while the view stayed put, with no re-sync. Fix: `routeFromHash` now re-asserts the hash to the actual `currentApp` after a switch attempt (reverts on a deferred/cancelled guard).
2. **[FIXED] No `beforeunload` guard.** Refresh / close-tab silently dropped unsaved note edits. Fix: a `beforeunload` listener triggers the browser's native "leave site?" prompt when `store.dirtyCheck()` is dirty (covers the notes editor — the main long-form risk). *Note:* the event-editor modal still isn't `beforeunload`-covered (it uses its own `kbDirty`, not `store.dirtyCheck`) — see Tier 3.
3. **[FIXED] Stale link-picker candidates.** `eventList` (notes `[[` picker) and `<linked-items>` sources were fetched once and cached for the session, so a freshly-created event/note couldn't be linked. Fix: refresh the event list on entering insert mode, and `<linked-items>` re-fetches its candidate lists each time the picker is focused.
4. **[FIXED] Failed lazy-load cached forever.** `loadScript` stored the rejected promise, so one failed chunk load (e.g. a network blip on `notes.js`/markdown-it) left that app permanently blank. Fix: delete the cache entry on error so the next visit retries.

## Tier 2 — latent multi-tenant gaps — **FIXED (this pass)**
5. **[FIXED] Notes vault is one shared directory.** `vault.ts` returned a single dir for all users; `scanVault` walked the whole vault and `scanFile` upserted by frontmatter `id` **without an owner filter**. Fix: per-owner vault root (`vault/<owner_id>/`) via `vaultRoot(owner)`; `abs(owner, relPath)` threads owner through every file op; `scanFile`'s existing-row lookup *and* update now filter `.where('owner_id','=',owner)` (a foreign same-id is a conflict, not an update). An idempotent boot migration (`migrateVaultLayout`, wired in `app.ts`) relocates any legacy flat files into their owner subdir. Covered by a new "per-owner: same title for two users doesn't collide or leak" test.
6. **[FIXED] Path/symlink hardening (defense-in-depth).** `abs()` now asserts the resolved path stays within the owner's vault root (`p === root || p.startsWith(root + sep)`, else throw); `walkMd` skips symlinks (`if (ent.isSymbolicLink()) continue`).
7. **[FIXED] Owner-scope the service reads.** `getNote`, `getEvent`, `updateEvent`, `archiveEvent` now take `owner` and push `.where('owner_id','=',owner)` into the query (on both the select and the update). All route callers updated to pass `request.user!.id`. Routes still guard with `denyAccess`, but the services are now safe-by-default if a future caller forgets.

### Open design — recurring-event occurrence expansion in unified search (#12 leftover) — **RESOLVED & IMPLEMENTED**

**Decisions made:** Option **A**'s occurrence-aware matching, but with **B**'s result identity — *one row per series* (deduped), dated to the soonest relevant occurrence, window today ± 1 year for open-ended predicates. Added a **`count`** field (matching occurrences in the window) so e.g. a thrice-weekly standup shows once as `count: 3` under `due:week` rather than three rows. Expansion triggers only when a `due:` term is present. Built in `unifiedQuery.ts` with a regression test. Original options/analysis kept below for context.


**The gap.** `runUnifiedQuery` maps each event to a single pseudo-task whose date is the **series start date** (`eventAsTask` → `due: startAt`). So date predicates (`due:today`, `due:week`, `before:…`, `after:…`, weekday windows) only ever test the *first* occurrence. A "weekly on Monday" standup that started in January matches `due:today` only on that first Monday — never again. The calendar grid is correct (`eventsInRange` expands occurrences via `Rec`); only the **search/`tdx-query`** path is not. Non-date predicates (free-text, `recurring:true`) are unaffected — this is purely about date matching on recurring series.

**Why it's not trivial.** Search has no inherent date window. Expanding a recurring series across *all time* is unbounded, so any expansion needs a window, and we have to decide what a "match" even returns (the series once, or each matching occurrence).

**Options (my read):**
- **A — Expand only when a date predicate is present, return matching occurrences.** If the query has any date term, expand each recurring series across a bounded window (mirroring `eventsInRange`) and emit one result item per occurrence that matches, each carrying its concrete `date` (like the calendar's `EventOccurrence`). With no date term, fall back to today's behavior (one item per series, start date). *Pro:* correct and intuitive — `due:today` lists exactly the events happening today, `due:week` the week's occurrences. Matches how the calendar already thinks. *Con:* a series can produce several rows; needs a window bound (suggest: clamp to the predicate's natural range, e.g. `due:week` → next 7 days; for open-ended ones like `due:set`/`before:` default to **today ± 1 year** and document the cap).
- **B — One row per series, predicate = "does ANY occurrence in the window match."** Keep results deduped to one item per event, but test the date predicate against the occurrence set instead of just the start date. *Pro:* no result explosion; stable row identity. *Con:* which date do you display? Ambiguous for `due:week` (which of several?), and it quietly diverges from the calendar's per-occurrence model.
- **C — Leave start-date semantics; just document it.** *Pro:* zero work, no surprises in result counts. *Con:* the feature stays subtly wrong; `due:today` keeps missing recurring events, which is exactly the kind of thing that erodes trust in search.

**My recommendation: A**, scoped to "expand only when the query carries a date predicate," with the window clamped to the predicate where possible and a documented ±1-year default otherwise. It's the only option that makes `due:today`/`due:week` actually trustworthy and it reuses `Rec`/`eventsInRange` logic we already have.
